# 学科知识整合智能体：基于混合 Function-Agent 架构的多教材知识图谱构建、选择式压缩与混合检索 RAG 系统

## 摘要

多教材知识整合是教育信息化中的核心挑战：教师需要将数百万字的教材内容压缩为可管理的精华版本，同时不能牺牲知识覆盖率、来源可追溯性和问答可信度。本文提出一个面向 7 本医学教材整合任务的全栈智能体系统。系统采用 Hybrid Function-Agent 架构，将确定性任务（解析、嵌入、检索）实现为纯函数，将需要自主决策的任务（跨教材整合、教师对话）实现为 Agent 模块。核心技术贡献包括：(1) 四策略级联解析器（PDF TOC → Markdown 标题 → 正则匹配 → 等量分页），在 7 本不同格式的教材上均成功提取章节结构；(2) 基于"原文精选"而非"全文改写"的压缩策略，将整合产物与 RAG 检索语料库解耦，保证引用 100% 真实；(3) 双阶段跨教材对齐算法（BGE-small-zh embedding 召回 + LLM 四要素精判 + Union-Find 聚类），实现知识点的传递性合并；(4) 向量 + BM25 混合检索与 Reciprocal Rank Fusion 融合排序的 RAG 管线。实验表明，7 本教材全量流程下系统将 832 个知识点整合为 288 个，压缩比 5.84%（远低于 30% 硬约束）；RAG 自建 10 题测试集上 keyword_hit@5 与 book_hit@5 均达 100%；四策略级联解析器在 7 本教材上策略匹配成功率 100%，平均每本解析耗时 12.4 秒。

**关键词**：知识图谱、跨教材整合、选择式压缩、混合检索 RAG、Hybrid Function-Agent 架构

## 1. 问题分析

### 1.1 任务定义

赛题要求构建一个学科知识整合智能体，能够：(1) 自动加载 7 本医学教材（PDF 格式，共约 826MB、326 万字）；(2) 为每本教材构建知识图谱并可视化；(3) 跨教材识别知识点的重叠、互补与缺失；(4) 将多本教材内容整合压缩到不超过原始体量 30% 的精华版本；(5) 基于整合后的知识库进行 RAG 精准问答，必须引用原文来源；(6) 支持教师通过多轮对话迭代优化整合方案。

### 1.2 核心挑战

经过深入分析，我们识别出四个核心技术挑战：

**挑战 1：章节结构提取的鲁棒性。** 7 本教材的 PDF 格式差异显著——部分含有 TOC 书签（如《生理学》），部分无 TOC 但有清晰排版（如《局部解剖学》），部分 TOC 中混杂 `\r` 等异常字符（如《病理学》）。传统的单策略解析方法无法覆盖所有情况。

**挑战 2：知识点颗粒度的统一。** 章节标题作为知识点过粗（如"血液"应拆为"红细胞""白细胞""血小板"等原子概念），句子片段过细。需要找到适合教学场景的中间颗粒度，使每个知识点可被"什么是 X？"的形式独立提问。

**挑战 3：压缩与引用真实性的矛盾。** 如果使用 LLM 全文改写来实现 30% 压缩，虽然可读性高，但会引入幻觉风险，破坏引用的真实性。估算 LLM 改写约 800 个知识点需消耗约 2.8M tokens，在免费 API 额度下不可行。

**挑战 4：RAG 的召回率与来源准确性。** 医学教材问答要求高召回率——遗漏关键信息可能导致误导。同时，回答中的每个事实必须附带准确的教材、章节和页码引用，不能用模型常识替代教材证据。

### 1.3 设计约束

在黑客松 5 小时的时间约束下，我们面临额外的工程限制：(1) 部署环境为 ModelScope Studio（2vCPU, 16GB 内存），不支持 GPU；(2) LLM 推理使用 ModelScope 免费 API 额度（DeepSeek-V3.2），存在限流风险；(3) 镜像大小受限，CUDA 版 PyTorch（~2GB）不可接受。

## 2. 方案设计

### 2.1 系统总体架构

系统采用 **Hybrid Function-Agent 架构**，核心设计理念是"按任务语义精确选型"——确定性任务用纯函数实现，需要自主决策的任务用 Agent 抽象。

```
┌───────────────────────────────────────────────────────┐
│                 前端层 (React + ECharts)                │
│  ┌──────────┐  ┌──────────────┐  ┌──────────────────┐ │
│  │ 上传面板  │  │  知识图谱视图 │  │ 整合/RAG/对话面板│ │
│  │          │  │  (力导向图)  │  │                  │ │
│  └──────────┘  └──────────────┘  └──────────────────┘ │
└───────────────────────┬───────────────────────────────┘
                        │ REST API
┌───────────────────────▼───────────────────────────────┐
│              后端层 FastAPI (Python 3.11)               │
│                                                       │
│  ┌─ 函数层（确定性模块）─────────────────────────┐    │
│  │ parser.py   → 四策略级联 PDF 解析器           │    │
│  │ chunker.py  → Markdown 感知滑动窗口切块       │    │
│  │ embedder.py → BGE-small-zh 嵌入编码器         │    │
│  │ retriever.py→ 向量 + BM25 + RRF 混合检索      │    │
│  │ extractor.py→ LLM JSON 知识点抽取             │    │
│  │ answerer.py → RAG 回答 + 引用生成             │    │
│  └───────────────────────────────────────────────┘    │
│                                                       │
│  ┌─ Agent 层（自主决策模块）─────────────────────┐    │
│  │ integrator.py → 整合 Agent                    │    │
│  │   (对齐 → 聚类 → 预算分配 → 决策输出)         │    │
│  │ dialog_agent.py → 教师对话 Agent               │    │
│  │   (意图路由 → 解释/修改/问答/摘要)            │    │
│  └───────────────────────────────────────────────┘    │
│                                                       │
│  ┌─ 数据层 ─────────────────────────────────────┐    │
│  │ store.py (SQLite)  │  嵌入缓存 (LRU)          │    │
│  │ 原始语料库 (chunks)│  压缩语料库              │    │
│  └───────────────────────────────────────────────┘    │
└───────────────────────────────────────────────────────┘
```

**架构设计论证：** 我们选择 Hybrid 而非纯 Agent 架构，基于以下考量：

| 阶段 | 任务性质 | 实现方式 | 论证 |
|------|---------|---------|------|
| 文档解析 | 确定性 I/O | 纯函数 | PDF 解析是规则驱动的，Agent 抽象只增加延迟和不确定性 |
| 嵌入/索引 | 确定性计算 | 纯函数 | 数值计算无需 LLM 介入 |
| 知识点抽取 | 单步 LLM 调用 | 纯函数 | 每章一次提示词 → 一次 JSON 输出，无需多步推理 |
| **跨教材整合** | **多步推理 + 工具调用** | **整合 Agent** | 需要自主选择合并/保留/移除，涉及预算约束和优先级排序 |
| **教师对话** | **多步推理 + 工具调用** | **对话 Agent** | 需要理解自然语言指令并路由到不同工具（解释决策、修改决策、知识问答、整合摘要） |
| RAG 检索+生成 | 单步管线 | 纯函数 | 标准"检索-生成"模式 |

这种精确选型使得系统在黑客松时间约束下能够快速实现和测试——每个纯函数模块可以独立单元测试，每个 Agent 可以通过重放日志验证行为。与"为拆 Agent 而拆"的做法相比，混合架构在可测试性、可调试性和开发速度上都有显著优势。

### 2.2 四策略级联解析器

解析器是系统的入口模块，其鲁棒性直接决定了下游所有模块的数据质量。我们设计了四级级联策略，确保任何 PDF 都能切出有意义的章节：

**Strategy 1 — PDF 内置 TOC（书签）：** 最准确的策略。读取 PDF 文件的书签树，过滤元页（封面、版权、前言、目录等 13 类关键词），优先选取含「章/篇/部分/Chapter/Part」关键字的 Level-1 条目。若 L1 含关键字的条目不足 3 个，回退到全部 L1；再不足则尝试 L2 含关键字的条目；最后尝试 L2 全部（3-50 条范围内）。

**Strategy 2 — Markdown 标题：** PyMuPDF4LLM 将 PDF 转为 Markdown 时，大字号文本会被标记为 H1/H2。此策略适用于无 TOC 但排版清晰的教材。优先按 H1 切分，H1 不足 3 个时回退到 H2。额外检查"单一巨块"情况——如果最大章节占总字符 85% 以上，判定切分失败，继续尝试下一策略。

**Strategy 3 — 文本正则匹配：** 使用严格的行首正则匹配「第X章」格式，要求标题独占一行并限定标题长度不超过 60 字符。此设计避免命中正文中的交叉引用（如"参见第六章）"这类非标题文本）。

**Strategy 4 — 等量分页兜底：** 当前三种策略均失败时，按每 30 页一段切分。每段独立调用 PyMuPDF4LLM 抽取 Markdown，保证页码元数据准确。

级联决策逻辑：Strategy 1 成功（章节数 ≥ 2）→ 使用 TOC；否则转 Markdown（策略 2/3 共用一次 Markdown 转换，避免重复 I/O），Strategy 2 章节数 ≥ 2 且非单一巨块 → 使用 MD headings；否则 Strategy 3 匹配数 ≥ 2 → 使用 regex；否则 Strategy 4 兜底。

**标题归一化：** 所有策略共享 `_norm()` 函数，统一处理 `\r\n`、全角空格 `　` 和末尾标点，解决了《病理学》等教材 TOC 中混杂 `\r` 导致标题匹配失败的问题。

### 2.3 知识图谱构建

#### 2.3.1 原子概念颗粒度

我们将知识点定义为 **2-12 字的原子名词短语**，可以被"什么是 X？"的形式独立提问。每章抽取 5-15 个知识点，按教学重要度排序。Prompt 明确禁止输出章节号本身、过粗的话题（如"血液"应拆为子概念）和含标点的复合短语。

#### 2.3.2 关系类型设计

系统定义了 4 种关系类型，覆盖教材知识的主要关联模式：

| 关系类型 | 语义 | 示例 |
|---------|------|------|
| `prerequisite` | A 是学习 B 的前置 | 静息电位 → 动作电位 |
| `parallel` | A 和 B 是同层级并列概念 | 红细胞 ↔ 白细胞 |
| `contains` | A 包含 B（上位/下位） | 免疫系统 → T 细胞 |
| `applies_to` | A 的应用场景是 B | 抗生素 → 感染性疾病 |

#### 2.3.3 抽取 Prompt 工程

抽取 prompt 采用以下设计：(1) **JSON 强约束**——system message 尾部追加"严格输出合法 JSON"；(2) **Few-shot 示例**——在 prompt 中给出完整的 nodes + edges JSON 示例，包含"动作电位""静息电位"等真实教材概念；(3) **后处理验证**——丢弃 source/target 未在 nodes 中声明的非法边、过滤含中文标点（`，。？！；：` 等）的脏名称、同章节内按 name 去重。

**超长章节处理：** 单章节 LLM 输入上限为 6,000 字符（约 3K token）。超长章节按 70%/30% 比例截取头尾——首段定义最密集，尾段常有总结。

#### 2.3.4 鲁棒性设计：启发式保底抽取

当 LLM API 不可用（额度耗尽、限流、超时）时，extractor 自动降级为启发式抽取：从章节内容中提取标题关键词、30+ 个医学高频术语（如"细胞""炎症""休克""水肿""发热""免疫""肿瘤"等）、Markdown 小标题中的名词短语，以及定义句式（"X 是指""X 包括""X 称为"）中的主语。启发式模块为每个候选术语从原文中截取包含该术语的完整句子作为定义，并生成 contains + parallel 两类基础关系。这确保了演示环境中即使 LLM 完全不可用，图谱仍能产出可视化节点和有意义的连接。

### 2.4 跨教材对齐与整合

#### 2.4.1 双阶段对齐算法

**Stage 1 — Embedding 召回：** 使用 BGE-small-zh-v1.5（512 维，L2 归一化后点积等价于余弦相似度）对"名称：定义前 200 字"文本编码，计算全量节点间的相似度矩阵。取上三角中 cosine ≥ 0.82 的对作为候选，同时**跳过同教材内的节点对**——同教材内的同名概念已在抽取阶段去重，此处只关注跨教材对齐。候选对按相似度降序排列。

**Stage 2 — LLM 精判（可选）：** 对候选对分批（每批 5 对）调用 LLM，按照四要素标准判定：

1. **指称同一对象**（无论中英文/同义词/缩写）
2. **核心定义重叠 ≥ 70%**
3. **学科范畴一致**
4. **抽象层级一致**（避免把上位概念和下位概念误判为相同，如"细胞" vs "神经细胞"）

LLM 输出 verdict ∈ {same, related, different}，附 confidence 和 reason。当 LLM 不可用时，系统降级为纯 embedding 对齐：cosine ≥ 0.92 的对自动合并。这一阈值设计基于经验观察——0.92 以上的候选对几乎都是精确同义词或完全相同的概念名。

**Stage 3 — Union-Find 聚类：** 将所有 verdict=same 且 confidence ≥ 0.7 的节点对输入 Union-Find 数据结构（路径压缩优化），实现传递性合并（A=B 且 B=C → A=C）。输出仅含 ≥ 2 个节点的簇。

#### 2.4.2 词法聚类增强

在语义对齐之外，系统额外执行词法聚类：对归一化后完全相同的节点名称（去除空格、标点、括号、大小写差异），如果分布在不同教材中则合并为一簇。词法聚类的结果与语义聚类通过第二轮 Union-Find 合并，确保既不遗漏精确匹配也不遗漏语义等价。

这一设计的动机是：embedding 相似度在短文本（2-12 字的知识点名称）上可能存在噪声，而完全相同的名称（如不同教材中的"细胞凋亡"）显然应该合并。词法聚类作为高精度补充，与语义对齐的高召回特性互补。

#### 2.4.3 选择式压缩策略

整合产物采用**原文精选 + 元数据导览**策略，而非 LLM 全文改写：

1. **Merge 决策：** 对每个聚类簇，选择重要度最高的节点作为 canonical 代表，重要度计算公式为：
   $$\text{importance}(n) = \text{degree}(n) \times 0.45 + \frac{\min(|\text{def}|, 220)}{220} \times 0.35 + \mathbb{1}[|\text{def}| \geq 12] \times 0.20$$
   保留 canonical 节点所在章节的原文摘录（≤ 900 字，以知识点名称为锚点定位上下文片段）。

2. **Keep 决策：** 对未被聚类的独有节点，按重要度降序排列，在 28% 预算（留 2% 缓冲）内逐个纳入，保留原文摘录（≤ 650 字）。

3. **Remove 决策：** 预算用尽后的剩余低重要度节点标记为 remove，附带理由说明。

**关键创新：双语料库分离。** 整合产物（≤ 30% 精华版本）和 RAG 检索语料库严格分离：

- **原始语料库**：原始教材文本块，始终作为 RAG 检索源。引用 100% 真实可跳页。
- **压缩语料库**：整合产物，仅供教师阅读和图谱审阅。

这一设计避免了 LLM 改写引入的幻觉风险，同时使压缩操作不影响 RAG 的召回质量。整合和检索两条管线完全独立，修改压缩策略不需要重建 RAG 索引。

### 2.5 RAG 检索增强生成管线

#### 2.5.1 Markdown 感知切块

chunker 采用三级切分策略：(1) 按 Markdown H2-H4 标题切为 section；(2) 每个 section 内用滑动窗口切块，目标长度 700 字，重叠 90 字；(3) 在窗口边界优先选择自然断点，优先级顺序为 `\n##` > `\n\n` > `。` > `；` > `\n`，避免在句子中间断裂。断点搜索要求位于窗口后 55% 的位置之后，确保每个 chunk 有足够的最小长度。

窗口大小选择 700 字的理由：700 字约合 350 个 token，加上 prompt 和回答空间后仍在 4K 上下文窗口内；同时每个 chunk 包含足够的上下文使单独阅读有意义。低于 80 字的零碎 chunk 被自动过滤。

#### 2.5.2 混合检索与 RRF 融合

检索阶段同时执行两路召回：

**向量检索路：** 使用 BGE-small-zh 对查询编码，与预计算的 chunk embedding 矩阵做点积（已 L2 归一化，等价于余弦相似度），取 top-30。embedding 在索引构建时预计算并以 binary blob 存入 SQLite，检索时反序列化为 numpy 数组。

**BM25 检索路：** 使用 jieba 分词构建语料库，BM25Okapi 实时打分，取 top-30。分词时过滤单字 token 和纯标点。BM25 对精确术语匹配（如"动作电位""静息电位"）有天然优势，弥补了向量检索在术语级精度上的不足。

**RRF 融合：** 采用 Reciprocal Rank Fusion（Cormack et al., 2009），将两路排名合并为统一分数：

$$\text{RRF}(d) = \sum_{r \in \{vec, bm25\}} \frac{1}{k + \text{rank}_r(d)}$$

其中 $k = 60$（标准参数）。最终取 RRF 分数 top-5 的 chunk 作为上下文输入给 LLM。

#### 2.5.3 严格引用提示词与降级策略

回答提示词要求：(1) 只基于给定上下文回答；(2) 每个关键事实后附 `[教材, 章节, 第X页]` 引用；(3) 上下文不足时回复"当前知识库中未找到相关信息"。

当 LLM 不可用时，系统降级为**抽取式回答**——直接展示 top-3 文本块的原文片段（截取 180 字）及来源信息，避免空响应。这一降级策略在 ModelScope 免费 API 额度用尽的场景下尤为关键，确保了演示现场的基本可用性。

### 2.6 教师对话 Agent

对话 Agent 实现了基于意图识别 + 工具调用的多轮对话：

| 用户意图 | 路由关键词 | 调用工具 | 行为 |
|---------|-----------|---------|------|
| 解释决策 | "为什么""解释""原因" | `explain_decision` | 在决策列表中模糊匹配，返回 reason 字段 |
| 修改决策（保留） | "保留""不要删除" | `modify_decision` | 将匹配的决策 action 更新为 keep |
| 修改决策（拆分） | "不要合并""拆开""不是同一个" | `modify_decision` | 将 merge 决策转为 keep |
| 知识问答 | "？""什么""如何""请问" | `query_rag` | 调用 RAG pipeline 回答 |
| 默认 | — | `summarize_integration` | 返回当前整合统计摘要 |

决策匹配支持两种方式：(1) 显式 decision_id 正则匹配（如 `run_xxx_merge_0001`）；(2) `[[知识点名]]` 双链语法或中文关键词在 reason/source_refs 中的模糊搜索。修改操作直接更新 SQLite 中的决策记录，后续查询即刻反映变化。聊天历史按 session_id 持久化，支持多轮上下文。

### 2.7 前端可视化

前端使用 React + ECharts 构建力导向图，支持以下交互功能：

- **节点大小映射：** `size = min(72, 15 + degree × 3 + click_count × 2)`，融合结构重要度和用户行为反馈
- **搜索高亮：** 实时过滤节点名/定义/章节/教材，匹配节点高亮（金色边框，`borderWidth=4`），未匹配节点透明度降至 0.18
- **全局 vs 单本图谱切换：** 全局模式按教材来源上色，单本模式按知识类别上色
- **节点详情侧抽屉：** 点击节点展示名称、定义、教材来源、章节、页码、出现频次和点击次数
- **教材原文溯源：** 从详情抽屉可加载对应章节原文，系统自动定位到包含该知识点名称的上下文片段（前后各 650 字窗口）
- **关系颜色编码：** prerequisite（红）、parallel（蓝）、contains（绿）、applies_to（紫），tooltip 显示关系类型和描述
- **三栏布局：** 左侧教材列表（上传/选择/删除）、中部图谱画布（全局/单本/章节三视图切换）、右侧功能面板（整合/RAG/对话三标签）

### 2.8 Obsidian 式双链

系统在整合报告中使用 `[[知识点]]` 作为双链表达，前端节点详情可显示教材来源、章节与相关节点。该设计把知识图谱与精华教材连接起来，使教师可以从图谱跳到原文证据，也可以从整合报告中的双链标记直接定位到图谱节点。

### 2.9 工程优化

#### 2.9.1 Docker 镜像瘦身

多阶段构建：第一阶段（Node.js 20 Alpine）编译前端，第二阶段（Python 3.11 slim）运行后端。关键优化是先安装纯 CPU 版 PyTorch（~200MB），再安装其他依赖，避免拉入 CUDA 版本（~2GB）。构建后清理测试目录和缓存文件，最终镜像约 1.8GB，满足 ModelScope 创空间的部署约束。

#### 2.9.2 LLM 多模型自动降级

LLM 客户端实现了自动降级链：主模型为 DeepSeek-V3.2，备选包括通过外部提供方调用的 DeepSeek-V3.2 和 XiaomiMiMo/MiMo-V2.5-Pro。当主模型遇到 429 限流、额度耗尽、服务不可用或模型未找到时，自动尝试下一个备选模型。JSON 解析采用宽松模式：先去除代码块包装，再从首个 `{` 或 `[` 截取到最后一个 `}` 或 `]`，兼容大多数 LLM 输出格式。

#### 2.9.3 并发控制

知识点抽取使用线程池（最大 5 并发）限制并发请求，避免 API 限流。跨教材对齐的 LLM 判定也使用相同的并发池。在 2 核 CPU 环境下，5 并发是 I/O 密集型任务的最优配置。

### 3.2 实验一：四策略级联解析器评估

我们在 7 本医学教材上测试了解析器的策略选择和章节提取效果：

| 教材 | 页数 | 命中策略 | 章节数 | 总字符数 | 耗时（秒） |
|------|-----:|---------|------:|--------:|--------:|
| 生理学 | 412 | pdf_toc (L1 关键字) | 13 | 298,461 | 14.2 |
| 局部解剖学 | 356 | pdf_regex | 22 | 245,830 | 11.8 |
| 传染病学 | 289 | pdf_regex | 22 | 201,456 | 9.7 |
| 微生物学 | 478 | pdf_toc (L2 全部) | 5 | 312,789 | 16.1 |
| 病理学 | 523 | pdf_toc (L1 归一化后) | 20 | 368,472 | 18.3 |
| 病理生理学 | 398 | pdf_toc (L1 关键字) | 20 | 285,106 | 12.0 |
| 药理学 | 345 | pdf_toc (L1 关键字) | 15 | 556,927 | 4.8 |
| **合计** | **2,801** | — | **117** | **2,269,041** | **86.9** |

**分析：**

- **策略匹配成功率 100%**：7 本教材均在 Strategy 1-3 中成功匹配，无一回退到 Strategy 4 兜底。
- **TOC 策略覆盖率 71.4%**（5/7）：大部分教材含有可用的 PDF 书签。
- **正则策略补充率 28.6%**（2/7）：对于无 TOC 的教材，行首正则有效识别"第X章"格式。
- **标题归一化修复**：病理学教材的 TOC 条目中混杂 `\r` 字符，归一化后成功匹配 20 个章节（修复前仅识别 1 个）。
- **平均解析速度**：12.4 秒/本，总计 86.9 秒完成全部 7 本教材的解析。
- **微生物学的二级目录回退**：该教材一级 TOC 条目仅有 1 个（不满足 ≥ 2 的阈值），系统自动回退到二级全部条目，成功识别 5 个章节。

### 3.4 实验三：跨教材整合效果

使用 7 本教材的全量知识图谱进行跨教材整合实验：

| 指标 | 结果 |
|------|-----:|
| 输入教材数 | 7 |
| 原始知识点 | 832 |
| 原始边数 | 737 |
| 整合后保留节点 | 288 |
| merge 决策数 | 20 |
| keep 决策数 | 268 |
| remove 决策数 | 0 |

**分析：**

- **压缩比 5.84%** 远低于 30% 硬约束，说明选择式压缩策略在预算控制上有较大余量。当前 28% 预算上限为 3,267,041 × 0.28 ≈ 914,772 字，而实际保留仅 190,930 字。
- **remove 决策为 0**：288 个保留节点的原文摘录总量未触及预算上限，因此所有通过对齐的节点均被保留。随着知识点质量提升和更精细的重要度排序，预期可以在保持低压缩比的同时收录更多详细原文。

#### 3.5.2 自建测试集评测

| 策略 | 估算令牌消耗 | 引用真实性 | 幻觉风险 | 实现复杂度 |
|------|---------------:|-----------|---------|-----------|
| LLM 全文改写 | 约 280 万 | 中（改写可能偏离原文） | 高 | 中 |
| LLM 摘要 + 引用验证 | 约 140 万 | 中高 | 中 | 高（需回查验证） |
| **原文精选 + 元数据**（本文） | **约 56 万** | **高（100% 原文）** | **无** | **低** |

**分析：** 原文精选方案相比 LLM 全文改写节省约 **5 倍**令牌成本（56 万 vs 280 万），且完全消除了幻觉风险。代价是精华版本的可读性依赖片段选择质量——如果选中的原文片段上下文不完整，阅读体验会打折扣。但在黑客松免费 API 额度的约束下，这一权衡是合理的。

## 4. 创新点总结

1. **Hybrid Function-Agent 架构**：不为 Agent 化而 Agent 化，按任务语义精确选型。确定性管线用纯函数保证可测试性和确定性，语义决策环节用 Agent 保证灵活性。这比全 Agent 架构更符合工程现实。

2. **四策略级联解析器**：覆盖有 TOC / 无 TOC / 有关键字标题 / 无结构信号的全部场景，实现 7/7 教材 100% 策略匹配。标题归一化统一处理编码异常，Markdown 转换在策略间共享避免重复 I/O。

3. **选择式压缩 + 双语料库分离**：整合产物用原文精选保证 100% 引用真实性，RAG 始终检索原始 chunk，两者解耦使压缩操作不影响问答质量。相比 LLM 改写方案节省 5 倍 token。

4. **双阶段对齐 + 词法增强**：embedding 语义召回 + LLM 四要素精判 + 词法精确匹配，三路证据通过 Union-Find 传递性合并，兼顾召回率和准确率。

5. **全链路降级保底**：LLM 不可用时，知识点抽取降级为规则式提取（医学术语库 + 句式匹配），RAG 回答降级为抽取式引文展示，对齐降级为高阈值 embedding 自动合并。保证系统在任何条件下都能产出有意义的结果。

## 5. 局限性与未来方向

### 5.1 当前局限

1. **RAG 评测规模**：自建测试集仅 10 题，不足以全面评估系统在不同问题类型上的表现。缺乏 MRR、NDCG 等标准检索指标。

2. **对齐精度评估**：当前缺乏人工标注的 ground truth，无法计算 merge 判定的 precision、recall 和 Cohen's kappa。部分教材因 LLM 限流使用启发式图谱兜底，后续应对低置信章节自动重抽。

3. **向量存储扩展性**：当前使用 SQLite 存储 embedding（binary blob），适合单机小规模场景。大规模教材库需迁移到 Chroma、FAISS 或 Milvus。

4. **图谱质量自动评估**：当前依赖人工检查，缺乏自动化的图谱质量指标（如知识覆盖率、关系准确率、与 MeSH/SNOMED CT 的对齐度）。

5. **压缩可读性**：原文精选的片段可能缺乏上下文连贯性，未来可在保留原文的基础上添加 LLM 生成的 1-2 句过渡语。

### 5.2 未来方向

1. **GraphRAG**：将知识图谱路径约束纳入 RAG 检索（Edge et al., 2024），例如"动作电位的前置知识是什么？"可以沿 prerequisite 边遍历，而不仅依赖向量相似度。

2. **增量更新**：当前整合是全量重算。未来可实现增量式——新增教材时仅对新节点执行对齐，而非重算全部相似度矩阵。

3. **教师反馈闭环**：当前对话 Agent 支持单次决策修改。未来可实现批量反馈（"这 5 个概念都不应该合并"）和自动学习教师偏好模式。

4. **多模态支持**：医学教材中图表（如解剖图、病理切片、分子结构图）含有关键教学信息。未来可集成多模态 LLM 将图表知识纳入图谱节点。

5. **评测体系完善**：引入外部医学知识库（如 MeSH、SNOMED CT）作为参考标准，自动评估知识图谱的覆盖率和准确率；构建 50+ 题的分层评测集覆盖事实、比较、推理和跨教材问题。

├── name: str (2-12 字原子概念)
├── definition: str (≤500 字)
├── category: str (核心概念/方法/定理/现象/...)
├── textbook_id / chapter_id: str
├── chapter_title: str
└── page: int

KnowledgeEdge
├── source / target: str (node id)
├── relation_type: prerequisite | parallel | contains | applies_to
└── description: str (≤200 字)

Chunk (RAG)
├── chunk_id: str (MD5-based)
├── textbook_id / chapter_id: str
├── textbook_title / chapter_title / section_title: str
├── page_start / page_end: int
├── text: str
└── char_count: int

IntegrationDecision
├── decision_id: str ({run}_{action}_{seq})
├── action: merge | keep | remove
├── affected_nodes: list[str]
├── result_node: str | None (canonical)
├── result_chunks: list[str]
├── source_refs: list[str]
├── source_excerpt: str (原文摘录)
├── reason: str
└── confidence: float
