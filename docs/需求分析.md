# 需求分析

## 摘要

本项目面向第一届 AI 全栈极速黑客松赛题：帮助教师把 7 本教材压缩整合为不超过原始体量 30% 的精华教材，同时保留原文可追溯性、知识图谱可解释性与 RAG 问答的引用可靠性。

系统目标不是简单摘要，而是完成“解析教材、构建知识图谱、跨教材对齐整合、生成精华版本、支持带引用问答、允许教师迭代修改”的端到端流程。

## 核心用户与场景

| 用户 | 目标 | 系统响应 |
| --- | --- | --- |
| 教师 | 快速理解多本教材的共同知识结构 | 上传教材后自动展示章节、知识点和关系 |
| 教师 | 判断哪些内容重复、哪些内容必须保留 | 输出 merge/keep/remove 决策及理由 |
| 教师 | 基于教材原文回答教学问题 | RAG 返回答案、教材名、章节与原文片段 |
| 教师 | 人工修正整合结果 | 对话 Agent 支持解释和修改整合决策 |
| 评委 | 复现实验与部署 | README、Docker、报告、测试集提供可验证路径 |

## 四个核心难点

| 难点 | 风险 | 设计要求 | 当前实现 |
| --- | --- | --- | --- |
| 知识颗粒度 | 过粗会变成章节摘要，过细会导致图谱噪声 | 2-12 字原子概念，优先名词短语 | `extractor.py` 用 few-shot JSON prompt 约束节点粒度 |
| 重复判定 | 同名不一定同义，异名可能同义 | 名称、定义、上下文、关系四要素共同判断 | embedding 召回 + LLM 精判 + Union-Find 聚类 |
| 30% 压缩 | LLM 重写会失真并增加成本 | 选择式压缩，保留原文摘录和来源 | `integrator.py` 按重要度选取原文片段，实测压缩比 7.1% |
| RAG 排序 | 单向量检索容易漏掉教材术语 | 章节感知切块，向量与 BM25 融合 | BGE embedding + BM25 + RRF，top-5 严格引用 |

## 功能性需求

| 编号 | 需求 | 验收口径 |
| --- | --- | --- |
| FR-1 | 支持 PDF、Markdown、TXT、DOCX 教材上传 | 能解析章节结构并在前端展示 |
| FR-2 | 为单本教材构建知识图谱 | 返回 JSON 节点和边，并可视化展示 |
| FR-3 | 支持跨教材整合 | 2 本以上教材自动识别重复知识点，输出决策列表 |
| FR-4 | 压缩比不超过 30% | 保留字符数 / 原始字符数 <= 0.3 |
| FR-5 | 支持 RAG 问答 | 回答必须带教材、章节、片段引用 |
| FR-6 | 支持多轮迭代 | 教师可用自然语言解释或修改整合决策 |
| FR-7 | 支持交互式图谱 | 缩放、移动、拖拽、点击详情、搜索、教材区分、频次映射 |

## 非功能性需求

| 维度 | 要求 | 当前策略 |
| --- | --- | --- |
| 可复现 | 本地和魔搭均可运行 | FastAPI + React + Docker + docker-compose |
| 成本控制 | 尽量减少外部 API 调用 | 本地 BGE embedding，LLM 只用于抽取、对齐和回答 |
| 鲁棒性 | LLM 或额度异常时不中断演示 | 多模型 fallback，抽取与回答均有启发式兜底 |
| 可追溯 | 所有精华内容可回到原教材 | chunk、节点和整合决策均保存教材、章节、页码元数据 |
| 可解释 | 整合原因可被审阅 | 决策列表包含 action、confidence、reason、affected_nodes |

## 知识图谱架构

系统采用三层知识图谱，而不是只做一个前端力导向图。

| 层级 | 内容 | 用途 |
| --- | --- | --- |
| 教材局部图谱 | 单本教材的 `KnowledgeNode` 与 `KnowledgeEdge` | 展示章节内知识结构，支持单本教材验收 |
| 跨教材对齐层 | 相似节点候选、LLM 判定、Union-Find cluster | 识别重复知识点与教材差异 |
| 整合产物层 | merge/keep/remove 决策、精华原文片段、引用来源 | 生成 <=30% 精华教材并供教师修改 |

节点字段包括：`id`、`name`、`definition`、`category`、`textbook_id`、`chapter_id`、`source_span`、`importance`、`frequency`、`click_count`。边字段包括：`source`、`target`、`relation`、`weight`、`reason`。

关系类型固定为四类：

| 关系 | 含义 |
| --- | --- |
| `prerequisite` | A 是理解 B 的前置知识 |
| `contains` | A 包含 B，适合章节或大概念分解 |
| `parallel` | A 与 B 属于并列概念 |
| `applies_to` | A 应用于 B，常见于方法、机制与场景 |

## Obsidian 式双链需求

本项目的双链不是独立笔记系统，而是对知识图谱的轻量表达层：

1. 整合报告和精华教材中用 `[[知识点]]` 标记可回链概念。
2. 前端点击节点时显示定义、所属教材、章节来源和相关节点。
3. RAG 引用块保留教材原文，避免双链跳转后只能看到模型改写文本。
4. 后续 Phase 7 全量 7 本教材完成后，可把 `[[知识点]]` 扩展为全局节点详情页。

## 分块策略选择依据

RAG 分块采用章节感知的 500-800 字窗口，保留 50-100 字 overlap。原因如下：

1. 教材 PDF 转 Markdown 后天然存在标题边界，优先按标题和段落切分能降低跨主题污染。
2. 600-700 字的中文 chunk 可以覆盖一个完整定义或机制解释，又不会让 top-5 上下文过长。
3. overlap 保留定义与例子之间的连续性，减少“定义在上一段、答案在下一段”的漏召回。
4. RAG 永远检索 `corpus_raw` 原文库，而不是整合后的精华版本，保证引用真实。

## Phase 7 全量目标

中期验证允许先用 2 本以上教材完成整合验收；最终目标必须更新为 7 本教材全量流程：

1. 上传并解析 7 本教材。
2. 为 7 本教材分别构建知识图谱。
3. 执行跨教材整合，压缩比保持在 30% 以内。
4. RAG 索引覆盖 7 本教材原文 chunk。
5. 图谱和整合报告显示 7 本教材来源分布。
